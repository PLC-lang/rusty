4//! ## [`Map`]
5//! This is a **simpler and faster** alternative implementation of the standard
6//! [`HashMap`][std::collections::HashMap] in **small-scale**. It does **not use
7//! heap** and **not need the [`Hash`][core::hash::Hash] or [`Ord`] trait** at all.
8//!
9//! It simply keeps all key-value pairs in an array and when it's necessary to
10//! retrieve by key, it scrolls through the entire array. This implementation
11//! works much faster for small maps of **less than 50 keys (recommended)**, but
12//! definitely is not suitable for larger maps.
13//!
14//! Check [this page](https://github.com/yegor256/micromap#benchmark)
15//! for the recent benchmarking results.
16//!
17//! For example, here is how a map with a few keys can be created:
18//! ```
19//! use micromap::Map;
20//! let mut map : Map<u64, &str, 10> = Map::new();
21//! map.insert(1, "Hello, world!");
22//! map.insert(2, "Good bye!");
23//! assert_eq!(map.len(), 2);
24//! assert_eq!(map.capacity(), 10);
25//! ```
26//!
27//! Creating a [`Map`] requires knowing the maximum size of it, upfront. This is
28//! what the third type argument `10` is for, in the example above. The array
29//! will have exactly ten elements. An attempt to [`insert`][Map::insert] an 11th
30//! element will lead to a panic. (Or use [`checked_insert`][Map::checked_insert]
31//! instead to avoid panics by returning an [`Option`].)
32//!
33//! ## [`Set`]
34//! Similarly, you can also create a **small [`Set`] on stack** directly. It has basic
35//! but adequate operations as a real set. (As is common practice, `Set<T>` is actually
36//! a `Map<T, ()>`)
37//!
38//! For example, here is how a set with a few items can be created and manipulated:
39//! ```
40//! use micromap::Set;
41//! let v = vec![5, 1, 2, 4];
42//! let set_a = Set::from([1, 2, 3]); // Set<i32, 3>
43//! let set_b: Set<i32, 4> = Set::from_iter(v);
44//! assert_eq!(set_a.capacity(), 3);
45//! assert_eq!(set_b.capacity(), set_a.capacity() + 1);
46//! // 5 or more (e.g. 7), or else panic
47//! let set_union: Set<_, 7> = set_a.union(&set_b).cloned().collect();
48//! let set_diff = &set_a - &set_b;
49//! assert_eq!(set_union, Set::from([1, 2, 3, 4, 5]));
50//! assert_eq!(set_diff, Set::from([3]));
51//! ```
52//! ## `no_std` support and Optional features
53//! The `micromap` does not depend on [`std`], so `no_std` is supported by default.
54//!
55//! And no feature is enabled by default:
56//! - `serde`: When this optional dependency is enabled, micromap implements the
57//!   `serde::Serialize` and `serde::Deserialize` traits.
58//! - `std`: Currently no effect, you can just ignore it.
59//!
60//! ## Capacity and Allocation
61//! The capacity **cannot be changed after creation** unless a new instance is created
62//! using a different generic constant. **No heap allocations** are made.
63//!
64//! If the type of the `Map` key-value pair (or the item type of the `Set`) is large, a
65//! stack-overflow error may occur when a large generic constant `N` is selected.
66//! Even if we use `Box::new(Set<BigT, 4096>)`, there is still a risk of stack-overflow
67//! unless Rust provides a usable Placement-New feature.
68//!
69//! Regarding memory usage, in addition to the inserted key-value pairs (or items),
70//! each [`Map`] or [`Set`] only occupies an additional `usize` of memory to store
71//! quantity (len) information, that's all.
