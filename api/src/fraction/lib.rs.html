3//! Fraction is designed to be a precise lossless drop-in replacement for floating types (f32, f64).
4//!
5//! It comes with a number of predefined type aliases covering the most common use cases such as
6//! [Fraction], [Decimal], [BigFraction], [DynaDecimal] and so on (see [prelude] module for more examples).
7//!
8//! The public API provides you with the generic types that you may use straightforwardly to build your
9//! own types, suiting your needs best (see [prelude] module for the examples).
10//!
11//! # Library features
12//!
13//! - Drop in replacement for floats with the exception for NaN == NaN so that it's hashable
14//! - It's hashable, so may be used as values in Sets and keys in dictionaries and hash maps
15//! - [Display](fraction::display) implementation for fractions and decimals
16//! - [Fraction](GenericFraction) type, representing fractions
17//! - [Decimal](GenericDecimal) type, based on [Fraction](GenericFraction) type represents floats as lossless decimals
18//! - [DynaInt](dynaint) implements dynamically growing integer type that perfarms checked math and avoids stack overflows
19//! - PostgreSQL binary protocol integration for both fractions and decimals
20//! - Juniper support for both fractions and decimals
21//! - [Generic integer conversions](generic), such as `i8 -> u8`, `usize -> u8` and so on
22//! - [Lossless division](division) with no allocations and infinite precision
23//!
24//! # Disclaimer
25//! Even though we do our best to keep it well covered with tests, there may be bugs out there.
26//! The library API is still in flux. When it gets stable we will release the version 1.0.0.
27//! You may find more info about Semantic Versioning on [https://semver.org/](https://semver.org/).
28//! Bug reports and contributions are appreciated.
29//!
30//! # Crate features
31//! - `with-bigint` (default) integration with [num::BigInt] and [num::BigUint] data types
32//! - `with-decimal` (default) [Decimal] type implemented upon [GenericFraction]
33//! - `with-dynaint` (default) dynamically growing integer avoiding stack overflows
34//! - `with-juniper-support` [Juniper](https://crates.io/crates/juniper) integration
35//! - `with-postgres-support` [PostgreSQL](https://crates.io/crates/postgres) integration; Numeric/Decimal type
36//! - `with-serde-support` [Serde](https://crates.io/crates/serde) traits implementation
37//!
38//! # Implementation
39//! Basic math implemented upon the [num] crate (in particular the [num::rational] module).
40//! The utilised traits from the [num] crate are re-exported, so you don't have to explicitly depend on that crate however,
41//! you may import them from either of crates if necessary.
42//!
43//! # Usage
44//! To start using types see the [Prelude](self::prelude) module.
45//!
46//! # Examples
47//!
48//! ## Simple use:
49//!
50//! ```
51//! type F = fraction::Fraction; // choose the type accordingly to your needs (see prelude module docs)
52//!
53//! let two = F::from(0) + F::from(2);   // 0 + 2 = 2
54//! let two_third = two / F::from(3);    // 2/3 = 0.666666[...]
55//!
56//! assert_eq!(F::from(2), two);
57//! assert_eq!(F::new(2u64, 3u64), two_third);
58//!
59//! assert_eq!("2/3", format!("{}", two_third));  // print as Fraction (by default)
60//! assert_eq!("0.6666", format!("{:.4}", two_third));  // format as decimal and print up to 4 digits after floating point
61//! ```
62//!
63//! Decimal is implemented as a representation layer on top of Fraction.
64//! Thus, it is also lossless and may require explicit control over "precision"
65//! for comparison and formatting operations.
66//! ```
67//! type D = fraction::Decimal;
68//!
69//! let result = D::from(0.5) / D::from(0.3);
70//!
71//! assert_eq!(format!("{}", result), "1.6"); // calculation result uses precision of the operands
72//! assert_eq!(format!("{:.4}", result), "1.6666");  // explicitly passing precision to format
73//!
74//! assert_eq!("1.6666", format!("{}", result.set_precision(4))); // the other way to set precision explicitly on Decimal
75//! ```
76//!
77//! ## Construct:
78//!
79//! Fraction:
80//! ```
81//! use std::str::FromStr;
82//! use fraction::{Fraction, Sign};  // choose the type accordingly with your needs (see prelude module docs)
83//!
84//! fn main() {
85//!     // There are several ways to construct a fraction, depending on your use case
86//!
87//!     let f = Fraction::new(1u8, 2u8);  // constructs with numerator/denominator and normalizes the fraction (finds least common denominator)
88//!     assert_eq!(f, Fraction::new_generic(Sign::Plus, 1i32, 2u8).unwrap());  // with numerator/denominator of different integer types
89//!     assert_eq!(f, Fraction::from(0.5));  // convert from float (f32, f64)
90//!     assert_eq!(f, Fraction::from_str("0.5").unwrap());  // parse a string
91//!
92//!     // Raw construct with no extra calculations.
93//!     // Most performant, but does not look for common denominator and may lead to unexpected results
94//!     // in following calculations. Only use if you are sure numerator/denominator are already normalized.
95//!     assert_eq!(f, Fraction::new_raw(1u64, 2u64));
96//! }
97//! ```
98//!
99//! Decimal:
100//! ```
101//! use std::str::FromStr;
102//! use fraction::{Decimal, Fraction};  // choose the type accordingly with your needs (see prelude module docs)
103//!
104//! fn main() {
105//!     // There are similar ways to construct Decimal. Underneath it is always represented as Fraction.
106//!     // When constructed, Decimal preserves its precision (number of digits after floating point).
107//!     // When two decimals are calculated, the result takes the biggest precision of both.
108//!     // The precision is used for visual representation (formatting and printing) and for comparison of two decimals.
109//!     // Precision is NOT used in any calculations. All calculations are lossless and implemented through Fraction.
110//!     // To override the precision use Decimal::set_precision.
111//!
112//!     let d = Decimal::from(1);  // from integer, precision = 0
113//!     assert_eq!(d, Decimal::from_fraction(Fraction::from(1))); // from fraction, precision is calculated from fraction
114//!
115//!     let d = Decimal::from(1.3);  // from float (f32, f64)
116//!     assert_eq!(d, Decimal::from_str("1.3").unwrap());
117//! }
118//! ```
119//!
120//! ## Format (convert to string)
121//! Formatting works similar for both Decimal and Fraction (Decimal uses Fraction internally).
122//! The format implementation closely follows the rust Format trait documentation.
123//!
124//! ```
125//! type F = fraction::Fraction;
126//!
127//! let result = F::from(0.7) / F::from(0.4);
128//! assert_eq!(format!("{}", result), "7/4");  // Printed as fraction by default
129//! assert_eq!(format!("{:.2}", result), "1.75"); // if precision is defined, printed as decimal
130//! assert_eq!(format!("{:#.3}", result), "1.750"); // to print leading zeroes, pass hash to the format
131//! ```
132//!
133//! ### Generic integer conversion
134//! ```
135//! use fraction::{Sign, GenericFraction};
136//!
137//! type F = GenericFraction<u32>;
138//!
139//! let fra = F::new_generic(Sign::Plus, 1i8, 42usize).unwrap();
140//! assert_eq!(fra, F::new(1u32, 42u32));
141//! ```
142//!
143//! ### Postgres usage
144//! Postgres uses i16 for its binary protocol, so you'll have to use at least u16
145//! as the base type for fractions/decimals.
146//! Otherwise you may workaround with DynaInt<u8, _something_more_than_u8_>.
147//! The safest way to go with would be DynaInt based types
148//! such as DynaFraction or DynaDecimal as they would prevent
149//! stack overflows for high values.
150//!
151//! Beware bad numbers such as 1/3, 1/7.
152//! Fraction keeps the highest achievable precision (up to 16383 digits after floating point).
153//! Decimal uses its own precision.
154//! So, if you may end up with bad numbers, it may be preferable to go with Decimals over Fractions.
155//!
156//! Both types (fractions and decimals) should work transparently
157//! in accordance with Postgres crate documentation
