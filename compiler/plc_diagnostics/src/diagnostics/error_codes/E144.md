# Invalid Expression in a CFC Variable

A variable in a CFC network (a data source or sink) carries its content as free text, and that text is
embedded into the compiled code as-is. Only plain *value* expressions are allowed — a name (`foo`), a
literal (`100`), member or array access (`foo[1].bar`), or compositions of them (`foo + 1`, `-bar`).

Everything else is rejected. Most importantly calls: in CFC a call must be drawn as a block, so the
network's evaluation priorities state explicitly *when* it runs. A call hiding inside a variable's text
(e.g. `foo()` or `foo() + 5`) would escape that ordering and would be re-evaluated at every consumer of
the variable, running its side effects an unpredictable number of times. The same rule also catches
text that is no expression at all, such as assignments (`a := b`), expression lists (`a, b`) or empty
variables.

Erroneous network:

```text
   conjure() + 5  ------->  result  (0)
```

To fix, draw the call as a block and wire its result instead:

```text
   +--- conjure ---+  (0)
   |        conjure|-------->[ + 5 ]------->  result  (1)
   +---------------+
```
